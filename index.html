🟢 ACID — Tranzaksiya Tamoyillari
🧠 Bu nima? ACID — PostgreSQL ma'lumotlar xavfsizligini kafolatlashi uchun 4 ta tamoyil. Har bir intervyuda 100% so'raladi!
🎯 Nima uchun kerak? Pul o'tkazish, buyurtma berish, inventar boshqarish — barchasi ACID ga tayanadi. Bu tamoyillarsiz ma'lumotlar buziladi.
📍 Qayerda? Har qanday database bilan ishlashda. ACID bu SQL ning asosi.
🔍 Chuqur tushuntirish
-- A = Atomicity  (Bo'linmaslik)  → Hammasi yoki hech narsa
-- C = Consistency (Izchillik)    → Qoidalar har doim saqlanadi
-- I = Isolation   (Izolyatsiya)  → Tranzaksiyalar bir-biriga xalaqit bermaydi
-- D = Durability  (Chidamlilik)  → COMMIT qilingan ma'lumot yo'qolmaydi
💻 A — Atomicity: Hammasi yoki hech narsa
-- Stsenariy: Ali dan Valiga 100,000 so'm o'tkazish
BEGIN;
  UPDATE accounts SET balance = balance - 100000 WHERE id = 1;  ◄--- Alidan ayirish
  UPDATE accounts SET balance = balance + 100000 WHERE id = 2;  ◄--- Valiga qo'shish
COMMIT;
-- Agar 2-UPDATE xato bersa → 1-UPDATE ham BEKOR bo'ladi!
-- Natija: Pul yo'qolmaydi, yarim holatda qolmaydi.
💻 C — Consistency: Qoidalar buzilmaydi
-- CHECK (balance >= 0) qo'yilgan bo'lsa:
BEGIN;
  UPDATE accounts SET balance = balance - 999999999 WHERE id = 1;
  -- ❌ CHECK buziladi → xatolik → ROLLBACK
  -- Database HECH QACHON noto'g'ri holatda qolmaydi
COMMIT;
💻 I — Isolation: Tranzaksiyalar alohida
-- 2 kishi bir vaqtda bitta tovarni sotib olmoqchi (stock=1):
-- Izolyatsiyasiz:
--   TX1: SELECT stock → 1 (bor)     TX2: SELECT stock → 1 (bor)
--   TX1: UPDATE stock=0             TX2: UPDATE stock=0
--   ❌ Muammo: 2 kishi sotib oldi, lekin stock faqat 1 edi!
-- Yechim: Isolation levels (67-bob) yoki FOR UPDATE (69-bob)
💻 D — Durability: COMMIT qilingan saqlanadi
-- COMMIT qilingan ma'lumot diskka yoziladi.
-- Server o'chib qolsa ham → WAL (Write-Ahead Log) dan tiklanadi.
-- WAL qanday ishlaydi:
--   1. O'zgarish avval WAL ga yoziladi (log fayl)
--   2. Keyin asosiy ma'lumotlar fayliga yoziladi
--   3. Server buzilsa → WAL dan tiklanadi (replay)

🟢 Isolation Levels — Izolyatsiya Darajalari
🧠 Bu nima? Bir vaqtda ishlayotgan tranzaksiyalar bir-birini qanchalik ko'rishini boshqarish. 4 ta daraja bor — har biri turli xil muammolardan himoya qiladi.
🎯 Nima uchun kerak? Noto'g'ri isolation = ma'lumot buzilishi. Masalan: siz o'qigan narx boshqa tranzaksiya tomonidan o'zgartirilgan bo'lishi mumkin!
⚙️ Sintaksis

BEGIN ISOLATION LEVEL REPEATABLE READ;             ◄--- Snapshot

-- 1. DIRTY READ: COMMIT qilinMAGAN ma'lumotni o'qish

-- 2. NON-REPEATABLE READ: bitta TX ichida 2 marta o'qish = farqli natija
--    TX1: SELECT price → 15M
--    TX2: UPDATE price=20M → COMMIT
--    TX1: SELECT price → 20M  ← FARQLI!

-- 3. PHANTOM READ: bitta TX ichida 2 marta COUNT = farqli natija
--    TX1: SELECT COUNT(*) → 7
--    TX2: INSERT qator → COMMIT
--    TX1: SELECT COUNT(*) → 8  ← YANGI qator paydo bo'ldi!
💻 READ COMMITTED (default)
BEGIN;  -- yoki BEGIN ISOLATION LEVEL READ COMMITTED
  SELECT price FROM product WHERE id = 1;      → 15000000
  -- (boshqa TX: UPDATE price=20M WHERE id=1 → COMMIT)
